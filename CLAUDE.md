## Правила работы

1. **Никогда не запускать ботов самостоятельно** - это делает только пользователь через Telegram
2. **Всегда деплоить изменения на сервер** после редактирования локальных файлов
3. После деплоя пользователь должен перезапустить ботов через Telegram
4. **Не забывать деплоить** - если изменил код локально, сразу залить на сервер
5. **НИКОГДА не деплоить папку profiles/** - настройки персонажей хранятся ТОЛЬКО на сервере и редактируются через Telegram бота. Локальные конфиги устарели!
6. **Коммиты на русском языке** - чтобы было понятно что изменилось
7. **После коммита всегда деплоим** - коммит + пуш + scp на сервер

## Технические детали кода

### Wicket AJAX Framework
Игра использует Apache Wicket с паттерном `Wicket.Ajax.get({u:'URL'})`.
- Парсинг через regex: `Wicket\.Ajax\.ajax\(\{[^}]*"c":"([^"]+)"[^}]*"u":"([^"]+)"`
- Заголовки для AJAX: `Wicket-Ajax: true`, `Wicket-Ajax-BaseURL`, `X-Requested-With`

### Metronome (Heartbeat)
**Критично для сбора лута!** Интервал 1.5 сек.
- Без heartbeat лут не собирается автоматически
- URL парсится из `Wicket.Timer.set('metronome', ...)`

### Combat System

## Создание персонажа (create_character.py)

**ВАЖНО:** Требуется **Node.js** на машине!

### Полный поток создания:
1. Главная `/` -> Нажать "Начать игру" -> `/training/battle` (лоадер)
2. Извлечь функцию `ppt()` и аргумент из HTML
3. Выполнить `ppt(arg)` через **node.js** -> получить `ppKey`
4. Вызвать `ppAction=go&ppKey=...` -> получить `urlStart`
5. Вызвать `urlStart&sex=m&side=1&type=1` -> персонаж создан -> таверна

### Почему нужен node.js:
Функция `ppt()` генерируется сервером с **уникальными константами для каждой сессии**. Нельзя захардкодить - нужно выполнять динамически.

### Ключевые находки:
- **URL страницы может быть `/training/battle`**, но контент - страница выбора класса! Проверять по **контенту**, не по URL
- После создания персонаж попадает в **ТАВЕРНУ**, не в бой
- Заголовок `ptxSPA: true` нужен для SPA-навигации
- `Ptx.Shadows.Tutor.urlStart` содержит базовый URL для создания
- Без `ppAction=go` сначала - получаем `accessDenied`

### Параметры создания:
| Параметр | Значения |
|----------|----------|
| sex | m (муж), f (жен) |
| side | 1 (свет), 2 (тьма) |
| type | 1 (Маг), 2 (Монах), 3 (Воин) |

## Исправленные баги

### Баг accessDenied при создании персонажа (2026-01-14)
**Проблема:** Сервер возвращал `accessDenied` при прямом запросе к `/user/customize?ppAction=select`.

**Причина:** Поток создания требует предварительного вызова `ppAction=go` с динамическим `ppKey`. Функция `ppt()` генерируется с уникальными константами для каждой сессии.

**Решение:**
1. Извлекаем `ppt()` и аргумент из HTML
2. Выполняем через node.js: `subprocess.run(['node', '-e', js_code])`
3. Вызываем `ppAction=go&ppKey=...`
4. Получаем `urlStart` и создаём персонажа

### Баг неработающего скилла в туториале (2026-01-14)
**Проблема:** Второй туториальный квест `new_tuturial2_quests_1_1` не сдавался (status: Fail). Требуется использовать скилл "Гром и Молния" в бою.

**Причина:** Метод `_find_skill_url()` искал паттерн `"c":"ptx_combat_rich2_skill_link_1"`, но HTML содержит другой формат: `"u":"...skills-0-skillBlock-skillBlockInner-skillLink..."`.

**Решение:** Добавлен новый паттерн поиска URL скилла по `skills-{index}-skillBlock-skillBlockInner-skillLink`. Также исправлен DEFAULT_BASE_URL с `vmmo.vten.ru` на `m.vten.ru` т.к. создание персонажа происходит на мобильном домене.

### Баг accessDenied в туториале (2026-01-14)
**Проблема:** TutorialRunner получал `accessDenied` при переходе в таверну после создания персонажа.

**Причина:** Создание персонажа происходит на `m.vten.ru`, куки привязаны к этому домену. TutorialRunner использовал `DEFAULT_BASE_URL = "https://vmmo.vten.ru"` - другой домен, куки не передаются.

**Решение:** Изменён `DEFAULT_BASE_URL` на `https://m.vten.ru` в `tutorial.py`.

### Баг "vmmo.vten.runone" в craft.py (2026-01-13)
**Проблема:** Крафт бронзового слитка падал с ошибкой DNS `vmmo.vten.runone`.

**Причина:** В `start_craft()` условие `level == 2` не покрывало бронзовый слиток (level=3). Код шёл в обычный путь где `f"{BASE_URL}{recipe['start_url']}"` склеивал домен с `None` → `"https://vmmo.vten.ruNone"`.

**Решение:** Изменено `level == 2` на `level >= 2` для всех рецептов 2+ уровня. Также добавлена валидация URL перед `urljoin()` во всех местах.

### Баг квест экипировки не сдаётся (2026-01-14)
**Проблема:** Третий туториальный квест `new_tuturial1_quests_1_2` (Основы экипировки) возвращал 404 при сдаче, хотя предмет был надет.

**Причина:** Код предполагал что `next_quest_id` из Complete API означает что квест автоматически принят. На самом деле это только **информация** о следующем квесте - сам квест НЕ принят! Сервер показывал `status=none, progress={current:0, total:1}`.

**Решение:**
1. Специальные квесты (equip_item, upgrade_amulet) ВСЕГДА требуют явного вызова `_accept_quest()` перед выполнением действия
2. Добавлена проверка статуса квеста через `apiFullUrl` перед попыткой Complete
3. Изменена логика: `if is_special or not next_quest_id: _accept_quest()`

**ВАЖНО для Claude:** `next_quest_id` из API - это только подсказка какой квест следующий, НЕ факт что он принят! Всегда проверять реальный статус через `apiFullUrl` или явно принимать квест.

### Баг "Работает: X ч" не сбрасывается после рестарта (2026-01-27)
**Проблема:** После перезапуска ботов у некоторых в веб-панели показывалось старое время работы (например "0.4 ч" вместо сброса).

**Причина:** `check_craft()` вызывался в `login()` ДО инициализации ресурсов в `run()`. При этом `check_craft()` переходил на страницы крафта, после чего `open_backpack()` мог вернуть не ту страницу, `parse_resources()` возвращал None и `start_session()` никогда не вызывался. В итоге `start_time` в resources.json оставался от предыдущей сессии.

**Решение:**
1. Создан метод `_init_resources_session()` в bot.py
2. Вызов перенесён в `login()` ПЕРЕД `check_craft()`
3. Удалён дублирующийся код из `run()`

**ВАЖНО для Claude:** Порядок инициализации критичен! Ресурсы должны парситься ДО любых действий, которые могут изменить текущую страницу.

### Баг ивент-данжены не работали из-за JS-рендеринга (2026-01-27)
**Проблема:** Ивент-данжены (Логово Демона Мороза, Пещеры Сурта и др.) не проходились несмотря на включённую настройку `ny_event_dungeon_enabled: true`. В логах: "Данжен dng:NYLairFrost_2026 не найден на странице".

**Причина:** Страница `/dungeons` рендерится JavaScript'ом. Python requests получает только HTML-скелет (~150KB) без данных о данженах. Бот искал `title="dng:NYLairFrost_2026"` в пустом HTML и всегда считал что данжен на КД.

**Решение:** Убрана проверка страницы `/dungeons`. Теперь `GenericEventDungeonClient.check_cooldown()` проверяет только кэш. Метод `enter_dungeon()` сразу идёт в lobby данжена (`/dungeon/lobby/NYLairFrost_2026?1=impossible`). Если на КД - сервер делает редирект на `/city` или `/dungeons`, что и определяет статус.

**ВАЖНО для Claude:** Многие страницы VMMO используют JS-рендеринг! Если бот не находит элементы которые видны в браузере - скорее всего это JS. Решение: идти напрямую по URL вместо парсинга списков.
